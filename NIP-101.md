# NIP-101

## Universal Lightning Wallet

`draft` `optional`

This NIP defines Universal Lightning Wallet (ULW), a Nostr-based protocol for connecting applications to Lightning wallets with scoped permissions, wallet-enforced spending limits and BOLT12 send/receive.

## Motivation

Applications need Bitcoin payment features without taking custody of user funds or implementing wallet-specific APIs.

NIP-47/Nostr Wallet Connect established pairwise app-wallet keys, encrypted requests over relays, and scoped wallet access. ULW uses the same base model, requires BOLT12 support, adds app-initiated authorization, requires wallet-enforced spending policy, and defines idempotent payments.

## Dependencies

Implementations use:

- NIP-01 for Nostr events, tags, filters, and signatures.
- NIP-40 for `expiration` tags.
- NIP-44 v2 for encrypted payloads.
- NIP-42 only when a chosen relay requires authenticated access.
- BIP-321 for payment instructions.
- BOLT11 invoices.
- BOLT12 offers.

## Event Kinds

This draft uses the following candidate event kinds:

| kind | name | use |
| --- | --- | --- |
| `19470` | `ULW_INFO` | replaceable wallet capability/info event |
| `39470` | `ULW_AUTH_RESPONSE` | app-initiated authorization response |
| `9470` | `ULW_REQUEST` | app request |
| `9471` | `ULW_RESPONSE` | wallet response |
| `9472` | `ULW_NOTIFICATION` | wallet notification |

Request, response, and notification events are regular stored events. Relays SHOULD store them until their `expiration` timestamp. Implementations MUST NOT rely on relays honoring expiration or deletion.

Every `ULW_REQUEST`, `ULW_RESPONSE`, and `ULW_NOTIFICATION` event MUST:

- be signed by the sender connection key;
- include exactly one `p` tag for the recipient connection key;
- include an `expiration` tag;
- include an `alt` tag;
- encrypt `content` using NIP-44 v2.

Responses MUST include an `e` tag referencing the request event.

`ULW_INFO` is an optional public replaceable event for non-user-specific service metadata. Its `content` is plaintext JSON:

```json
{
  "version": "ulw-1",
  "methods": ["get_info", "get_balance", "pay", "receive", "make_invoice", "make_offer", "lookup_payment", "list_transactions", "disconnect"],
  "authorization": ["app_initiated", "wallet_initiated"]
}
```

`ULW_INFO` MUST NOT contain user-specific wallet data. `get_info` is authoritative for a specific connection.

## Connections

A connection is one app-wallet authorization. It is identified by a pairwise app connection key and wallet connection key.

Apps and wallets SHOULD use a fresh connection key for each authorization. They MUST NOT use a user's long-term public Nostr identity key as a connection key unless the user explicitly chooses that privacy loss.

Connection material MUST include one or more relay URLs. Implementations SHOULD use at least two independent relays when practical. Senders SHOULD publish to all connection relays; receivers SHOULD subscribe on all connection relays and deduplicate events.

Relays are untrusted. They can drop, delay, reorder, duplicate, censor, and store events.

## Authorization

ULW supports app-initiated and wallet-initiated authorization.

### App-Initiated Authorization

The app generates:

- a fresh app connection keypair;
- a random `state` value with at least 128 bits of entropy;
- required commands;
- optional commands;
- an optional budget;
- relay URLs where it will listen for the response.

The app presents:

```text
nostr+ulw-auth://<app_pubkey>?relay=<url>&state=<hex>&required=<commands>&optional=<commands>&budget=<base64url-json>&name=<app-name>&expires_at=<unix>
```

Authorization URI fields:

- `<app_pubkey>`: REQUIRED. App connection public key as lowercase hex.
- `relay`: REQUIRED and repeatable. Relay URLs where the app listens for the response.
- `state`: REQUIRED. Lowercase hex with at least 128 bits of entropy.
- `required`: REQUIRED. Space-separated command names.
- `optional`: OPTIONAL. Space-separated command names.
- `budget`: OPTIONAL. Base64url-encoded budget JSON.
- `name`: OPTIONAL. Human-readable app name.
- `expires_at`: OPTIONAL, except REQUIRED when a provided budget does not include `window_seconds`. Unix timestamp.

The wallet MUST show the user the requesting app, requested commands, requested budget if present, expiry, and relay list before approval. The wallet MAY grant fewer commands than requested. It MUST NOT grant commands the app did not request unless the user is explicitly shown them.

If approved, the wallet publishes kind `39470` encrypted to the app pubkey:

```json
{
  "type": "authorization_response",
  "version": "ulw-1",
  "state": "same-random-state",
  "approved": true,
  "relays": ["wss://relay.example"],
  "permissions": {
    "commands": ["get_balance", "pay", "receive"],
    "notifications": {
      "types": ["payment_received", "payment_sent"],
      "delivery": "stored"
    }
  },
  "budget": {
    "max_amount_msat": 1000000,
    "window_seconds": 86400,
    "max_payment_msat": 250000,
    "max_fee_msat": 5000,
    "max_fee_ppm": 1000
  },
  "expires_at": 1790000000
}
```

The event author is the wallet connection pubkey. The event MUST include a `p` tag for the app pubkey, a `d` tag equal to `state`, and an `expiration` tag. The app MUST verify the event author, NIP-44 decryption, `state`, relay list, granted commands, and budget if present.

Approved authorization response fields:

- `type`: REQUIRED. `authorization_response`.
- `version`: REQUIRED. `ulw-1`.
- `state`: REQUIRED. Same value from the authorization URI.
- `approved`: REQUIRED. `true`.
- `relays`: REQUIRED. Relay URLs for the connection.
- `permissions`: REQUIRED. Granted commands and optional notification permissions.
- `budget`: OPTIONAL. Wallet-enforced budget for this connection.
- `expires_at`: REQUIRED. Unix timestamp for grant expiry.

If rejected, the wallet MAY publish an authorization response with `approved: false` and an error object.

### Wallet-Initiated Authorization

A wallet MAY generate a NWC-like bearer URI:

```text
nostr+ulw://<wallet_pubkey>?relay=<url>&secret=<app_private_key_hex>&commands=<commands>&budget=<base64url-json>&expires_at=<unix>
```

Wallet-initiated URI fields:

- `<wallet_pubkey>`: REQUIRED. Wallet connection public key as lowercase hex.
- `relay`: REQUIRED and repeatable. Relay URLs for the connection.
- `secret`: REQUIRED. App connection private key as lowercase hex.
- `commands`: REQUIRED. Space-separated granted command names.
- `budget`: OPTIONAL. Base64url-encoded budget JSON.
- `expires_at`: REQUIRED. Unix timestamp for grant expiry.

The `secret` is spend-authority material. Anyone who sees it can use the grant until expiry or revocation, subject to wallet-enforced permissions, any budget, and local policy.

Apps MUST store wallet-initiated authorization secrets in protected storage.

### Local Provisioning

An application using a local or embedded wallet MAY provision an equivalent connection locally instead of using an authorization URI.

Local provisioning is outside this NIP and is defined by the relevant wallet SDK.

A wallet SDK that claims ULW compatibility should expose the same request, response, command, budget, idempotency, and error semantics defined by this NIP.

## Permissions

Permissions are granted per connection. A wallet MUST reject any command not granted to the connection.

Defined commands:

- `get_info`
- `get_balance`
- `pay`
- `receive`
- `make_invoice`
- `make_offer`
- `lookup_payment`
- `list_transactions`
- `disconnect`

`pay` is a spend command. Future spend commands MAY use a wallet-enforced budget.

`receive` is the generic receive command. `make_invoice` and `make_offer` are explicit receive commands for apps that need to request a specific Lightning payment instruction.

## Budgets

Spend-authorized connections MAY have a wallet-enforced budget. App-side limits are not a security boundary.

If present, a budget object MUST include `max_amount_msat`. It MUST include either `window_seconds` or an authorization `expires_at`.

```json
{
  "max_amount_msat": 1000000,
  "window_seconds": 86400,
  "max_payment_msat": 250000,
  "max_fee_msat": 5000,
  "max_fee_ppm": 1000
}
```

`max_amount_msat` is REQUIRED because it bounds total outgoing liability. It is the maximum cumulative outgoing debit for the budget window or authorization lifetime. Wallets MUST count settled outgoing amount plus fees against it.

`window_seconds` is OPTIONAL if the authorization has `expires_at`. When a budget is present, at least one of `window_seconds` or `expires_at` is REQUIRED so the budget has a bounded duration.

`max_payment_msat`, `max_fee_msat`, and `max_fee_ppm` are OPTIONAL but RECOMMENDED. `max_fee_msat` is an absolute fee cap. `max_fee_ppm` is a proportional fee cap. If both are present, wallets MUST enforce both. A wallet MUST reject a payment before attempting it if the amount or maximum fee can exceed any provided budget or per-payment limit.

Wallets MAY enforce stricter local policies.

## Request and Response Format

All encrypted plaintext MUST be UTF-8 JSON and MUST include `version: "ulw-1"`.

Request plaintext:

```json
{
  "type": "request",
  "version": "ulw-1",
  "request_id": "random-128-bit-or-larger-id",
  "method": "pay",
  "params": {}
}
```

Response plaintext:

```json
{
  "type": "response",
  "version": "ulw-1",
  "request_id": "same-id",
  "result_type": "pay",
  "result": {},
  "error": null
}
```

`request_id` MUST be unique for each logical request. Apps MUST reuse the same `request_id` when retrying the same logical request. Wallets MUST make spend commands idempotent by requester connection pubkey and `request_id`. A duplicate spend request MUST NOT execute a second payment.

The event `expiration` tag defines request expiry. A wallet MUST reject an expired request.

## Errors

When applicable, wallets MUST use these error codes:

| code | meaning |
| --- | --- |
| `BAD_REQUEST` | malformed request or invalid parameters |
| `UNAUTHORIZED` | unknown or revoked app key |
| `RESTRICTED` | command not granted |
| `QUOTA_EXCEEDED` | budget exceeded |
| `INSUFFICIENT_BALANCE` | insufficient spendable balance |
| `PAYMENT_FAILED` | payment attempted and failed |
| `NOT_FOUND` | requested record not found |
| `EXPIRED` | request expired |
| `DUPLICATE` | duplicate request cannot return prior result |
| `NOT_IMPLEMENTED` | known method not implemented |
| `UNSUPPORTED_PAYMENT_INSTRUCTION` | unsupported BIP-321 instruction |
| `UNSUPPORTED_NETWORK` | unsupported Bitcoin network |
| `RATE_LIMITED` | app is sending too many requests |
| `TEMPORARILY_UNAVAILABLE` | wallet cannot process now; app may retry |
| `INTERNAL` | wallet internal failure |
| `OTHER` | failure not covered above |

## Commands

### `get_info`

Returns wallet and connection capabilities.

Result fields:

- `alias`
- `network`: `mainnet`, `testnet`, `signet`, or `regtest`
- `methods`: granted methods for this connection
- `notifications`: notification types granted to this connection and their delivery mode
- `budget`: OPTIONAL. Wallet-enforced budget for this connection

`methods` MUST describe this connection, not merely global wallet support.

### `get_balance`

Returns:

- `balance_msat`
- `available_msat`

`available_msat` SHOULD be immediately spendable balance after wallet reserves.

### `pay`

Pays a BIP-321 payment instruction.

Params:

- `payment`: REQUIRED BIP-321 URI
- `amount_msat`: REQUIRED if the selected instruction has no amount
- `max_fee_msat`: REQUIRED unless the wallet applies a stricter connection policy
- `max_fee_ppm`: OPTIONAL
- `payer_note`: OPTIONAL
- `timeout_seconds`: OPTIONAL
- `metadata`: OPTIONAL

ULW wallets MUST support BIP-321 URIs containing one or more of these Lightning payment instructions:

- `lightning`: BOLT11 invoice
- `lno`: BOLT12 offer

A payment URI MAY contain `lightning`, `lno`, or both. If more than one supported instruction is present, the wallet MUST choose one supported instruction, pay only that instruction, and report the paid instruction in `instruction_type`.

Other BIP-321 instructions are not required. Unsupported instructions MUST be rejected before payment.

For BOLT12 offers, the wallet MUST perform invoice request and invoice retrieval internally.

If `payer_note` is present and the selected payment instruction supports payer-provided messages, the wallet MUST deliver it. If the wallet cannot deliver a non-empty `payer_note`, it MUST reject the request before payment.

Wallets MUST validate network compatibility and MUST NOT exceed `max_fee_msat`, `max_fee_ppm`, connection budget if present, or local policy.

Result fields:

- `payment_id`
- `state`
- `instruction_type`: `bolt11` or `bolt12`
- `amount_msat`
- `fees_paid_msat`
- `payment_hash`, if available
- `preimage`, if available
- `payer_proof`: BOLT12 `lnp` payer proof, if available
- `txid`: on-chain transaction id, if available
- `created_at`
- `settled_at`, if settled

### `receive`

Creates a BIP-321 receive URI.

Params:

- `amount_msat`: OPTIONAL; omitted or null requests a variable-amount receive URI
- `description`: OPTIONAL

Result fields:

- `bip321`: BIP-321 URI containing one or more wallet-selected receive instructions
- `receive_id`: wallet-scoped id for the receive target, if available
- `payment_id`: wallet-scoped payment id, if available

The wallet chooses the receive instruction or instructions encoded in `bip321`. Apps that need a specific Lightning instruction should use `make_invoice` or `make_offer`.

### `make_invoice`

Creates a BOLT11 invoice.

Params:

- `amount_msat`
- `description` or `description_hash`
- `expiry_seconds`
- `metadata`

Result fields:

- `payment_id`
- `bip321`: MUST contain a `lightning` instruction
- `invoice`: raw BOLT11 invoice, OPTIONAL
- `payment_hash`
- `amount_msat`
- `created_at`
- `expires_at`
- `state`

### `make_offer`

Creates or returns a BOLT12 offer usable as a receive target.

Params:

- `amount_msat`: OPTIONAL; omitted or null requests a variable-amount offer
- `description`
- `issuer`
- `single_use`
- `expires_at`
- `metadata`

Wallets MUST support reusable variable-amount offers. Wallets MAY return an existing offer matching the requested parameters.

Result fields:

- `offer_id`
- `bip321`: MUST contain an `lno` instruction
- `offer`: raw BOLT12 offer, OPTIONAL
- `amount_msat`
- `single_use`
- `created_at`
- `expires_at`

A wallet MUST keep BOLT12 offers returned by `make_offer` payable while the user-facing application is offline. This requires an always-online wallet service or async payments. Interactive-only BOLT12 receive is not sufficient.

### `lookup_payment`

Looks up a payment by `payment_id`, `payment_hash`, invoice, or offer id.

Result fields:

- `payment_id`
- `type`: `incoming` or `outgoing`
- `state`
- `instruction_type`
- `amount_msat`
- `fees_paid_msat`
- `payment_hash`, if available
- `preimage`, if available
- `payer_proof`: BOLT12 `lnp` payer proof, if available
- `txid`, if available
- `payer_note`, if available
- `created_at`
- `settled_at`, if settled
- `metadata`

### `list_transactions`

Lists transactions visible to this connection.

Params:

- `from`
- `until`
- `limit`
- `cursor`
- `type`: `incoming`, `outgoing`, or omitted

Wallets SHOULD return newest first. Wallets SHOULD use opaque cursors. `limit` MUST be capped at 20 unless a higher limit is explicitly negotiated.

Transaction records use the same fields as `lookup_payment`. They MUST include inbound and outbound payment message fields, including `payer_note`, when available.

### `disconnect`

Revokes the current app connection.

After revocation, the wallet MUST reject requests from the revoked app key even if relays continue to deliver old events. Relay deletion MUST NOT be relied on for security.

## Payment States

Payment records MUST use one of:

- `pending`
- `settled`
- `failed`
- `expired`
- `canceled`

## Notifications

Notifications are optional state-change hints. Apps MUST NOT require notifications for correctness; they must reconcile with `lookup_payment` or `list_transactions`.

Defined notification types:

- `payment_received`
- `payment_sent`
- `authorization_changed`

Defined notification delivery modes:

- `stored`: the wallet can publish notification events to connection relays without the app being online.
- `deferred`: the wallet publishes notification events only when the wallet is next available.

Notification plaintext:

```json
{
  "type": "notification",
  "version": "ulw-1",
  "notification_id": "wallet-stable-id",
  "notification_type": "payment_received",
  "notification": {}
}
```

Wallets MUST NOT advertise real-time notification support unless some wallet service can publish notifications while the user-facing app is offline. Mobile-only wallets MAY advertise deferred notifications.

## Metadata

`metadata` fields are optional opaque JSON objects. Apps MUST NOT place secrets, access tokens, seed material, or Nostr identity keys in metadata.

Wallets MAY store metadata with payment records. Wallets MUST reject or drop metadata larger than 4096 UTF-8 bytes.

Wallets and apps MUST sanitize invoice descriptions, offer descriptions, payer notes, and metadata before display.

## Security Considerations

ULW grants are bearer capabilities. Possession of the app connection private key is sufficient to exercise the grant until expiry or revocation, subject to wallet-enforced permissions, any budgets, and local policy.

NIP-44 does not provide forward secrecy. Compromise of a connection private key can decrypt past stored messages for that connection. Implementations SHOULD rotate connection keys and use short expiry for spend permissions.

Relays see metadata: event kinds, timestamps, authors, recipients in tags, event sizes, and network-level information. ULW does not hide app-wallet relationship metadata from relays.

Provided budgets MUST be enforced by the wallet. App-side checks are not a security boundary.

Spend commands MUST be idempotent. Without idempotency, relay replay and app retry can duplicate payments.
