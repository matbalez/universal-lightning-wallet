# NIP-102

## BOLT12 Zaps

`draft` `optional`

This NIP defines BOLT12 zaps: public Nostr events proving that a BOLT12 payment was made to the author of a profile, event, or addressable event.

BOLT12 zaps replace the LNURL callback flow used by NIP-57 with BOLT12 offers and payer proofs. No LNURL endpoint, zap callback server, separately relayed zap request event, or recipient-operated receipt publisher is required.

## Motivation

NIP-57 made Lightning payments visible on Nostr, but it depends on LNURL pay servers. A sender asks the recipient's server for a BOLT11 invoice, pays it, and then trusts that server to publish a zap receipt.

BOLT12 changes the shape of the problem. A recipient can publish a reusable offer. A payer can request an invoice over the Lightning network, pay it, receive an `lnp` payer proof, and publish a Nostr event proving the payment.

The goal of this NIP is simple: a Nostr event can accumulate a client-verifiable total amount of Bitcoin zapped to it, paid directly to the target recipient's BOLT12 offer.

## Dependencies

Implementations use:

- NIP-01 for events, tags, filters, and signatures.
- NIP-100 for profile-level BOLT12 offer discovery.
- NIP-19 and NIP-21 where clients display Nostr entity references.
- BOLT12 offers, invoices, and `lnp` payer proofs.

## Event Kind

This draft uses two candidate event kinds:

| kind | name | use |
| --- | --- | --- |
| `9736` | `bolt12_zap` | public BOLT12 payment proof |
| `9738` | `bolt12_zap_intent` | signed pre-payment zap intent embedded in `9736` |

Kind `9736` is the only event counted as a BOLT12 zap.

## BOLT12 Zap Targets

Profile-level BOLT12 zap recipients are discovered using NIP-100.

The canonical raw offer is the NIP-100 `lno` value: the lowercase `lno1...` BOLT12 offer string with BOLT12 `+` separators and whitespace removed.

An event MAY override or split zap recipients with `zap12` tags:

```json
["zap12", "<recipient-pubkey>", "<relay-url>", "<weight>", "<lno>"]
```

Fields after `<recipient-pubkey>` are OPTIONAL. If `<lno>` is absent, clients SHOULD resolve the recipient's profile-level offer using NIP-100, using `<relay-url>` as a relay hint when present.

If `<lno>` is present, it MUST be a canonical raw BOLT12 offer.

If an event has no `zap12` tag, the default zap recipient is the event author and the offer is resolved using NIP-100.

For durable event-level validation, event authors SHOULD include the offer in a `zap12` tag. Relying only on mutable profile metadata can make old zap validation fail after offer rotation.

For split zaps, clients compute recipient shares from tag weights using the same weighting rule as NIP-57 `zap` tags. Each recipient payment produces a separate kind `9736` zap event.

## Zap Intent

Before paying, the payer creates and signs a zap intent event. The zap intent binds the payer's Nostr key to the recipient, target, amount, and BOLT12 offer before the Lightning payment is attempted.

The payer chooses a `zap_id` with at least 128 bits of entropy, encoded as lowercase hex.

A zap intent event is a signed event of kind `9738`.

Required tags:

- `p`: recipient pubkey.
- `amount`: amount in millisatoshis.
- `offer`: canonical raw BOLT12 offer.
- `zap_id`: random lowercase hex value.

Optional tags:

- `e`: event being zapped.
- `a`: addressable event coordinate being zapped.
- `k`: kind of the event being zapped.

`content` MAY contain a zap comment.

The zap intent event MUST have exactly one `p` tag. It MUST have at most one `e` tag and at most one `a` tag, and MUST NOT contain both `e` and `a`.

Anonymous zaps use an ephemeral event pubkey.

The payer MUST include the zap intent event id as the BOLT12 `invreq_payer_note`:

```text
nostr:nip102:<zap-intent-event-id>
```

## Zap Event

A zap event is a signed event of kind `9736`.

Required tags:

- `description`: serialized zap intent event.
- `p`: recipient pubkey.
- `amount`: amount in millisatoshis.
- `offer`: canonical raw BOLT12 offer.
- `proof`: BOLT12 `lnp` payer proof.

Optional tags:

- `P`: payer pubkey. If present, it MUST equal the zap event `pubkey`.
- `e`: event being zapped.
- `a`: addressable event coordinate being zapped.
- `k`: kind of the event being zapped.

The `description` tag uses the same embedding pattern as NIP-57. Its value MUST be the complete serialized kind `9738` zap intent event as JSON.

`content` MUST equal the zap intent event `content`.

Example:

```json
{
  "kind": 9736,
  "content": "excellent note",
  "tags": [
    ["description", "{\"kind\":9738,\"content\":\"excellent note\",\"tags\":[[\"p\",\"32e182...e245\"],[\"e\",\"9ae37a...94fb\"],[\"k\",\"1\"],[\"amount\",\"21000\"],[\"offer\",\"lno1...\"],[\"zap_id\",\"4f1c8f6a0d5e4a4e94f5ef99d1c1e8ad\"]],\"pubkey\":\"97c70a...e322\",\"created_at\":1781699999,\"id\":\"d8f3...9c1a\",\"sig\":\"f4b1...7a02\"}"],
    ["p", "32e182...e245"],
    ["P", "97c70a...e322"],
    ["e", "9ae37a...94fb"],
    ["k", "1"],
    ["amount", "21000"],
    ["offer", "lno1..."],
    ["proof", "lnp1..."]
  ],
  "pubkey": "97c70a...e322",
  "created_at": 1781700012
}
```

The zap event MUST have exactly one `description` tag and exactly one `p` tag. It MUST have at most one `e` tag and at most one `a` tag, and MUST NOT contain both `e` and `a`.

Publicly attributed zaps SHOULD include `P`. Anonymous zaps MUST use an ephemeral event pubkey and MUST omit `P`.

The zap event `pubkey`, `content`, `p`, `amount`, `offer`, `e`, `a`, and `k` presence and values MUST match the embedded zap intent event.

## Payer Proof Disclosure

The `proof` tag MUST contain a bech32-encoded BOLT12 `payer_proof` with human-readable prefix `lnp`.

For NIP-102 validation, the payer proof MUST disclose at least:

- `invreq_payer_id`;
- `invreq_payer_note`;
- `invoice_payment_hash`;
- `invoice_amount`;
- `invoice_node_id`;
- `signature`;
- `proof_preimage`;
- `proof_signature`;
- every offer TLV present in the zap event's `offer` tag that is mirrored into the invoice request or invoice;
- `invoice_features`, if present.

The proof MUST NOT omit any field required by NIP-102 validation behind `proof_omitted_tlvs`.

`invreq_payer_note` MUST equal:

```text
nostr:nip102:<zap-intent-event-id>
```

The x-only form of `invreq_payer_id` MUST equal the zap event `pubkey` and the embedded zap intent event `pubkey`.

The proof MUST disclose enough offer, invoice request, and invoice fields for verifiers to confirm that the paid invoice was derived from the canonical raw BOLT12 offer in the zap event and embedded zap intent. A proof that cannot be validated against that exact offer MUST NOT be counted as a BOLT12 zap.

## Payment Flow

To send a BOLT12 zap:

1. Resolve the target recipient and BOLT12 offer.
2. Choose the Nostr key that will publish the zap event.
3. Choose a random `zap_id`.
4. Create and sign a kind `9738` zap intent event.
5. Pay the offer for the requested amount, including `nostr:nip102:<zap-intent-event-id>` in BOLT12 payer metadata.
6. Obtain a settled `lnp` payer proof from the payer wallet.
7. Publish a kind `9736` zap event carrying the serialized zap intent event and the proof.

## Async Payments

A kind `9736` zap event MUST prove settled payment.

Pending, queued, held, async-handoff, or invoice-request-only artifacts MUST NOT be published as kind `9736` zaps and MUST NOT be counted as verified zaps.

If an async payment flow does not produce settled payer proof until the recipient or recipient infrastructure comes online, publication of the zap event MUST wait until that proof exists.

This sacrifices immediate zap publication for offline recipients, but it prevents a payer from making an unverifiable claim such as a very large pending payment and having clients count it as a valid zap.

## Validation

A client MUST validate a zap event before counting it.

Validation steps:

1. Verify the zap event signature.
2. Verify the event has kind `9736`.
3. Verify the event has exactly one `description` tag, exactly one `p` tag, a positive `amount`, a canonical raw BOLT12 `offer`, and an `lnp` `proof`.
4. Verify the event has at most one `e` tag and at most one `a` tag, and does not contain both.
5. If `P` is present, verify it equals the event `pubkey`.
6. Parse the `description` tag as a Nostr event.
7. Verify the zap intent event signature and verify it has kind `9738`.
8. Verify the zap intent event `pubkey` equals the zap event `pubkey`.
9. Verify the zap event `content`, `p`, `amount`, `offer`, `e`, `a`, and `k` values match the zap intent event.
10. Verify the zap intent event has exactly one `p` tag, a positive `amount`, a canonical raw BOLT12 `offer`, a valid `zap_id`, at most one `e` tag, at most one `a` tag, and does not contain both `e` and `a`.
11. Parse the raw BOLT12 offer from the `offer` tag.
12. Verify the offer is authorized by the recipient:
   - it appears in the target event's signed `zap12` tag, or
   - it matches the recipient's NIP-100 public BOLT12 receive offer, using relay hints from the relevant `zap12` tag when present.
13. Decode and validate the `lnp` payer proof according to the BOLT12 payer-proof specification.
14. Verify `SHA256(proof_preimage)` equals `invoice_payment_hash`.
15. Reconstruct the invoice Merkle root and verify `signature` using `invoice_node_id`.
16. Verify `proof_signature` using `invreq_payer_id`.
17. Verify the x-only form of `invreq_payer_id` equals the zap event `pubkey`.
18. Verify `invreq_payer_note` equals `nostr:nip102:<zap-intent-event-id>`.
19. Verify `invoice_amount` equals the event `amount`.
20. Verify the payer proof is valid for the canonical raw BOLT12 offer in the zap event and embedded zap intent.
21. Verify every offer TLV required by this NIP is disclosed and matches the zap event's raw offer.

If no settled `lnp` payer proof is available, the event MUST NOT be counted as a BOLT12 zap.

## Counting Zaps

To compute the verified cumulative zap amount for an event, clients query for kind `9736` events with `#e` equal to the event id. For addressable events, clients query by `#a`. For profile zaps, clients query by `#p` and ignore events that also contain `e` or `a`.

Clients MUST validate each zap event before counting it.

Clients MUST deduplicate zap events for the same target by proof identifier. The proof identifier is `invoice_payment_hash` from the validated payer proof.

If two valid zap events for the same target have the same proof identifier, clients MUST count only one. If amounts differ, clients MUST count the lower amount.

The cumulative zap amount is a client-computed view over valid public proof events. It is not consensus.

## Privacy

BOLT12 zaps are public proof events. They reveal at least amount, target, recipient pubkey, event time, and relays used. If `P` is present, they also reveal payer identity.

Payers who want private payments SHOULD pay the BOLT12 offer without publishing a zap event. Private payments are not counted as public zaps.

BOLT12 blinded paths can hide the recipient's node id, but Nostr zap metadata still reveals social payment relationships.

## Non-Goals

This NIP does not define:

- LNURL zap callbacks;
- BOLT11 zap invoices;
- custodial zap servers;
- private zaps;
- on-chain zaps;
- Cashu or Nutzap receipts;
- anti-sybil scoring policy;
- relay-level aggregation;
- pending or claimed-but-unsettled zaps.

## Rationale

A separately relayed zap request event is unnecessary when the signed zap intent event is embedded in the final zap event. Verifiers still get the important NIP-57 property: the payer signed the Nostr zap context before payment.

The zap intent event binds the payer's Nostr key to the zap target. The BOLT12 payer note binds the Lightning payment proof to that signed intent. Without this binding, a payer could reuse proof from one payment to claim zaps on unrelated events.

Only settled payer proofs count. Counting pending async payment artifacts would let payers fake arbitrary zap amounts.

## Compatibility

NIP-102 does not replace NIP-57 immediately. Clients MAY display NIP-57 and NIP-102 receipts together, but they MUST validate and label them separately.

NIP-57 `zap` tags are not reused. Events SHOULD use `zap12` tags for BOLT12 zap splits to avoid ambiguity with LNURL zaps.
