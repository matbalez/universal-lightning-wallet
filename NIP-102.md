# NIP-102

## BOLT12 Zaps

`draft` `optional`

This NIP defines BOLT12 zaps: public Nostr receipts for BOLT12 payments made to the author of a profile, event, or addressable event.

BOLT12 zaps replace the LNURL callback flow used by NIP-57 with BOLT12 offers and payment proofs. No LNURL endpoint, zap callback server, or new external server is required.

## Motivation

NIP-57 made Lightning payments visible on Nostr, but it depends on LNURL pay servers. A sender asks the recipient's server for a BOLT11 invoice, pays it, and then trusts that server to publish a zap receipt. The receipt is not a strong payment proof; it mostly proves that some server said an invoice was paid.

BOLT12 changes the shape of the problem. A recipient can publish a reusable offer. A payer can request an invoice over the Lightning network, pay it, and receive proof material from the payment flow. Nostr only needs to carry the signed zap intent and the public receipt.

The goal of this NIP is simple: a Nostr event can accumulate a client-verifiable total amount of Bitcoin zapped to it, paid directly to the target recipient's BOLT12 offer.

## Dependencies

Implementations use:

- NIP-01 for events, tags, filters, and signatures.
- NIP-19 and NIP-21 where clients display Nostr entity references.
- BIP-321 for `bitcoin:` payment instructions.
- BOLT12 offers, invoices, and payer proofs.
- NIP-101 where wallets expose BOLT12 payment and proof material to applications.

NIP-57 is not required. LNURL is not used.

## Event Kinds

This draft uses the following candidate event kinds:

| kind | name | use |
| --- | --- | --- |
| `9736` | `bolt12_zap_request` | signed zap intent |
| `9737` | `bolt12_zap_receipt` | public zap receipt |

A `bolt12_zap_request` MAY be unpublished. If published, clients MUST NOT count it as a payment.

A `bolt12_zap_receipt` is the only event counted as a BOLT12 zap.

## BOLT12 Zap Targets

A user who wants to receive BOLT12 zaps SHOULD publish a BOLT12 offer in their kind `0` metadata:

```json
{
  "name": "alice",
  "lno": "lno1..."
}
```

The `lno` value MAY be a raw BOLT12 offer or a BIP-321 URI containing an `lno` instruction.

An event MAY override or split zap recipients with `zap12` tags:

```json
["zap12", "<recipient-pubkey>", "<relay-url>", "<weight>", "<lno-or-bip321>"]
```

Fields after `<recipient-pubkey>` are OPTIONAL. If `<lno-or-bip321>` is absent, clients SHOULD fetch the recipient's kind `0` metadata from `<relay-url>` or from normal metadata relays.

If an event has no `zap12` tag, the default zap recipient is the event author and the offer is resolved from that author's kind `0` metadata.

For durable event-level validation, event authors SHOULD include the offer in a `zap12` tag. Relying only on mutable profile metadata can make old receipts harder to validate after offer rotation.

For split zaps, clients compute recipient shares from tag weights using the same weighting rule as NIP-57 `zap` tags.

## Zap Request

A zap request is a signed event of kind `9736`. It binds the payer, recipient, target, amount, offer, and optional comment.

Required tags:

- `p`: recipient pubkey.
- `amount`: amount in millisatoshis.
- `offer`: BIP-321 URI containing an `lno` instruction, or a raw BOLT12 offer.
- `relays`: relays where the receipt should be published.

Optional tags:

- `e`: event being zapped.
- `a`: addressable event coordinate being zapped.
- `k`: kind of the event being zapped.

`content` MAY contain a zap comment.

Example:

```json
{
  "kind": 9736,
  "content": "excellent note",
  "tags": [
    ["p", "32e182...e245"],
    ["e", "9ae37a...94fb"],
    ["k", "1"],
    ["amount", "21000"],
    ["offer", "bitcoin:?lno=lno1..."],
    ["relays", "wss://relay.example", "wss://relay2.example"]
  ],
  "pubkey": "97c70a...e322",
  "created_at": 1781700000
}
```

The zap request MUST have exactly one `p` tag. It MUST have at most one `e` tag and at most one `a` tag. It MUST have a valid Nostr signature.

Clients SHOULD NOT publish zap requests before payment. Publishing them before payment leaks intent and creates spam without proving value transfer.

## Payment Flow

To send a BOLT12 zap:

1. Resolve the target recipient and BOLT12 offer.
2. Create and sign a kind `9736` zap request.
3. Pay the offer for the requested amount.
4. Bind the payment to the zap request by including the zap request id in the payer-provided note or equivalent BOLT12 payer metadata:

```text
nostr:nip102:<zap-request-id>
```

5. Obtain proof material from the payer wallet.
6. Publish a kind `9737` zap receipt.

Wallets implementing NIP-101 can perform step 3 with `pay` using a BIP-321 `lno` instruction and `payer_note: "nostr:nip102:<zap-request-id>"`.

## Zap Receipt

A zap receipt is a signed event of kind `9737`.

Required tags:

- `p`: recipient pubkey.
- `amount`: amount in millisatoshis.
- `request`: JSON-encoded zap request event.
- `proof`: BOLT12 payer proof.

Optional tags:

- `P`: payer pubkey. REQUIRED for public attribution. MUST be omitted for anonymous zaps.
- `e`: event being zapped.
- `a`: addressable event coordinate being zapped.
- `k`: kind of the event being zapped.
- `payment_hash`: payment hash, if available.
- `preimage`: payment preimage, if available.
- `offer`: BOLT12 offer or BIP-321 URI used.

Example:

```json
{
  "kind": 9737,
  "content": "",
  "tags": [
    ["p", "32e182...e245"],
    ["P", "97c70a...e322"],
    ["e", "9ae37a...94fb"],
    ["k", "1"],
    ["amount", "21000"],
    ["request", "{\"kind\":9736,...}"],
    ["proof", "lnp1..."],
    ["payment_hash", "hex-if-available"]
  ],
  "pubkey": "97c70a...e322",
  "created_at": 1781700012
}
```

The canonical payer pubkey is the embedded zap request `pubkey`. The receipt event `pubkey` MUST equal the embedded zap request `pubkey`.

If `P` is present, it MUST equal both the receipt event `pubkey` and the embedded zap request `pubkey`.

Publicly attributed zaps MUST include `P`. Anonymous zaps MUST be created and published from an ephemeral pubkey and MUST omit `P`.

## Receipt Validation

A client MUST validate a zap receipt before counting it.

Validation steps:

1. Verify the receipt event signature.
2. Parse and verify the embedded zap request from the `request` tag.
3. Verify the request has kind `9736`, a valid signature, exactly one `p` tag, and a positive `amount`.
4. Verify the receipt event `pubkey` equals the embedded request `pubkey`.
5. If `P` is present, verify it equals the receipt event `pubkey` and embedded request `pubkey`.
6. Verify receipt `p`, `e`, `a`, `k`, and `amount` tags match the embedded request.
7. Verify the request offer is authorized by the recipient:
   - it appears in the target event's signed `zap12` tag, or
   - it appears in the recipient's signed kind `0` metadata, or
   - it appears in signed metadata referenced by the target event's `zap12` tag.
8. Verify the BOLT12 payer proof.
9. Verify the proof commits to the request id, preferably through the BOLT12 payer note value `nostr:nip102:<zap-request-id>`.
10. Verify the proof amount is equal to the request and receipt `amount`.

When a BOLT12 payer proof is present, clients MUST validate it according to the BOLT12 payer-proof specification. The proof MUST reveal enough information to verify:

- the payment amount;
- the payment hash or equivalent proof identifier;
- the invoice was derived from the recipient's offer;
- the payer metadata commits to the zap request id.

If no BOLT12 payer proof is available, clients MAY verify `preimage` against `payment_hash`, but such receipts are settlement-only receipts. They MUST NOT be included in verified cumulative zap totals. They SHOULD be displayed differently and SHOULD NOT be used for spam deterrence, ranking, or any workflow requiring payer attribution.

## Counting Zaps

To compute the verified cumulative zap amount for an event, clients query for kind `9737` events with `#e` equal to the event id. For addressable events, clients query by `#a`.

Clients MUST validate each receipt before counting it.

Clients MUST deduplicate receipts for the same target by proof identifier. For verified zap totals, the proof identifier is the hash of the BOLT12 payer proof.

If two valid receipts for the same target have the same proof identifier, clients MUST count only one. If amounts differ, clients MUST count the lower amount.

The cumulative zap amount is a client-computed view over valid public receipts. It is not consensus.

## Privacy

BOLT12 zaps are public receipts. They reveal at least amount, target, recipient pubkey, receipt time, and relays used. If `P` is present, they also reveal payer identity.

Payers who want private payments SHOULD pay the BOLT12 offer without publishing a zap receipt. Private payments are not counted as public zaps.

BOLT12 blinded paths can hide the recipient's node id, but Nostr receipt metadata still reveals social payment relationships.

## Non-Goals

This NIP does not define:

- LNURL zap callbacks;
- BOLT11 zap invoices;
- custodial zap servers;
- private zaps;
- on-chain zaps;
- Cashu or Nutzap receipts;
- anti-sybil scoring policy;
- relay-level aggregation.

## Rationale

The zap request remains a signed Nostr event because it cleanly binds payer intent to a Nostr target.

The zap receipt is public because zaps are social proof. Clients can sum receipts without asking the recipient's server.

BOLT12 offers remove the need for LNURL invoice callbacks. A recipient publishes a reusable offer, and the payer's wallet performs the invoice request over Lightning.

The payment must commit to the zap request id. Without this binding, a payer could reuse proof from one payment to claim zaps on unrelated events.

Payer proofs are preferred over preimage-only receipts because a preimage alone does not prove payer identity and can be replayed by anyone who sees it.

## Compatibility

NIP-102 does not replace NIP-57 immediately. Clients MAY display NIP-57 and NIP-102 receipts together, but they MUST validate and label them separately.

NIP-57 `zap` tags are not reused. Events SHOULD use `zap12` tags for BOLT12 zap splits to avoid ambiguity with LNURL zaps.
