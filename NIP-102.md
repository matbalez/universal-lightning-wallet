# NIP-102

## BOLT12 Zaps

`draft` `optional`

This NIP defines BOLT12 zaps: public Nostr events proving that a BOLT12 payment was made to the author of a profile, event, or addressable event.

BOLT12 zaps replace the LNURL callback flow used by NIP-57 with BOLT12 offers and payer proofs. No LNURL endpoint, zap callback server, zap request event, or recipient-operated receipt publisher is required.

## Motivation

NIP-57 made Lightning payments visible on Nostr, but it depends on LNURL pay servers. A sender asks the recipient's server for a BOLT11 invoice, pays it, and then trusts that server to publish a zap receipt.

BOLT12 changes the shape of the problem. A recipient can publish a reusable offer. A payer can request an invoice over the Lightning network, pay it, receive payer proof material, and publish a Nostr event proving the payment.

The goal of this NIP is simple: a Nostr event can accumulate a client-verifiable total amount of Bitcoin zapped to it, paid directly to the target recipient's BOLT12 offer.

## Dependencies

Implementations use:

- NIP-01 for events, tags, filters, and signatures.
- NIP-19 and NIP-21 where clients display Nostr entity references.
- BIP-321 for `bitcoin:` payment instructions.
- BOLT12 offers, invoices, and payer proofs.
- NIP-101 where wallets expose BOLT12 payment and proof material to applications.

NIP-57 is not required. LNURL is not used.

## Event Kind

This draft uses one candidate event kind:

| kind | name | use |
| --- | --- | --- |
| `9737` | `bolt12_zap` | public BOLT12 payment proof |

Kind `9737` is the only event counted as a BOLT12 zap.

## BOLT12 Zap Targets

A user who wants to receive BOLT12 zaps SHOULD publish a BOLT12 offer in their kind `0` metadata using the `lno` field defined by NIP-101:

```json
{
  "name": "alice",
  "lno": "bitcoin:?lno=lno1..."
}
```

The `lno` value MAY be a raw BOLT12 offer or a BIP-321 URI containing an `lno` instruction. Senders MUST normalize a raw offer to BIP-321 before paying through a NIP-101 wallet.

An event MAY override or split zap recipients with `zap12` tags:

```json
["zap12", "<recipient-pubkey>", "<relay-url>", "<weight>", "<lno-or-bip321>"]
```

Fields after `<recipient-pubkey>` are OPTIONAL. If `<lno-or-bip321>` is absent, clients SHOULD fetch the recipient's kind `0` metadata from `<relay-url>` or from normal metadata relays.

If an event has no `zap12` tag, the default zap recipient is the event author and the offer is resolved from that author's kind `0` metadata.

For durable event-level validation, event authors SHOULD include the offer in a `zap12` tag. Relying only on mutable profile metadata can make old zap validation fail after offer rotation.

For split zaps, clients compute recipient shares from tag weights using the same weighting rule as NIP-57 `zap` tags. Each recipient payment produces a separate kind `9737` zap event.

## Zap Commitment

Before paying, the payer constructs a zap commitment. The commitment binds the payment proof to the Nostr target without requiring a separate zap request event.

The payer chooses a `zap_id` with at least 128 bits of entropy, encoded as lowercase hex.

The payer computes:

```text
commitment = sha256(zap_commitment_string)
```

where `zap_commitment_string` is UTF-8 text with exactly these lines and newline separators:

```text
nip102-zap-v1
zap_id=<zap_id>
payer=<payer-pubkey>
recipient=<recipient-pubkey>
amount_msat=<decimal-msat>
target=<profile|event:<event-id>|address:<a-coordinate>>
offer_sha256=<sha256-of-raw-bolt12-offer>
```

`payer` is the pubkey that will sign the kind `9737` zap event. Anonymous zaps use an ephemeral pubkey.

`offer_sha256` is the SHA-256 hash of the raw BOLT12 offer string, after extracting it from the BIP-321 `lno` instruction if necessary.

The payer MUST include the commitment in the BOLT12 payer-provided note or equivalent payer metadata:

```text
nostr:nip102:<commitment-hex>
```

## Zap Event

A zap event is a signed event of kind `9737`.

Required tags:

- `p`: recipient pubkey.
- `amount`: amount in millisatoshis.
- `offer`: BIP-321 URI containing the BOLT12 offer in an `lno` instruction.
- `zap_id`: random lowercase hex value used in the zap commitment.
- `proof`: BOLT12 payer proof.

Optional tags:

- `P`: payer pubkey. If present, it MUST equal the zap event `pubkey`.
- `e`: event being zapped.
- `a`: addressable event coordinate being zapped.
- `k`: kind of the event being zapped.
- `payment_hash`: payment hash or equivalent proof identifier, if available.

`content` MAY contain a zap comment.

Example:

```json
{
  "kind": 9737,
  "content": "excellent note",
  "tags": [
    ["p", "32e182...e245"],
    ["P", "97c70a...e322"],
    ["e", "9ae37a...94fb"],
    ["k", "1"],
    ["amount", "21000"],
    ["offer", "bitcoin:?lno=lno1..."],
    ["zap_id", "4f1c8f6a0d5e4a4e94f5ef99d1c1e8ad"],
    ["proof", "lnp1..."],
    ["payment_hash", "hex-if-available"]
  ],
  "pubkey": "97c70a...e322",
  "created_at": 1781700012
}
```

The zap event MUST have exactly one `p` tag. It MUST have at most one `e` tag and at most one `a` tag.

Publicly attributed zaps SHOULD include `P`. Anonymous zaps MUST use an ephemeral event pubkey and MUST omit `P`.

## Payment Flow

To send a BOLT12 zap:

1. Resolve the target recipient and BOLT12 offer.
2. Choose the Nostr key that will publish the zap event.
3. Choose a random `zap_id`.
4. Compute the zap commitment.
5. Pay the offer for the requested amount, including `nostr:nip102:<commitment-hex>` in BOLT12 payer metadata.
6. Obtain settled payer proof material from the payer wallet.
7. Publish a kind `9737` zap event carrying the proof.

Wallets implementing NIP-101 can perform step 5 with `pay` using a BIP-321 `lno` instruction and `payer_note: "nostr:nip102:<commitment-hex>"`. The wallet must return payer proof material for the application to publish a proof-grade zap.

## Async Payments

A kind `9737` zap event MUST prove settled payment.

Pending, queued, held, async-handoff, or invoice-request-only artifacts MUST NOT be published as kind `9737` zaps and MUST NOT be counted as verified zaps.

If an async payment flow does not produce settled payer proof until the recipient or recipient infrastructure comes online, publication of the zap event MUST wait until that proof exists.

This sacrifices immediate zap publication for offline recipients, but it prevents a payer from making an unverifiable claim such as a very large pending payment and having clients count it as a valid zap.

## Validation

A client MUST validate a zap event before counting it.

Validation steps:

1. Verify the zap event signature.
2. Verify the event has kind `9737`.
3. Verify the event has exactly one `p` tag, a positive `amount`, a valid `zap_id`, an `offer`, and a `proof`.
4. Verify the event has at most one `e` tag and at most one `a` tag.
5. If `P` is present, verify it equals the event `pubkey`.
6. Extract the raw BOLT12 offer from the `offer` tag.
7. Verify the offer is authorized by the recipient:
   - it appears in the target event's signed `zap12` tag, or
   - it appears in the recipient's signed kind `0` metadata, or
   - it appears in signed metadata referenced by the target event's `zap12` tag.
8. Recompute the zap commitment from the event `pubkey`, `zap_id`, recipient, target, amount, and offer hash.
9. Verify the BOLT12 payer proof.
10. Verify the proof commits to `nostr:nip102:<commitment-hex>` through BOLT12 payer metadata.
11. Verify the proof amount equals the event `amount`.
12. Verify the proof is for a settled payment and that the invoice was derived from the recipient's offer.
13. If the proof identifies a payer pubkey, verify it equals the zap event `pubkey`.

When a BOLT12 payer proof is present, clients MUST validate it according to the BOLT12 payer-proof specification. The proof MUST reveal enough information to verify:

- the payment amount;
- the payment hash or equivalent proof identifier;
- the invoice was derived from the recipient's offer;
- the payer metadata commits to the zap commitment;
- the payment is settled.
- the payer identity, if exposed by the proof, matches the zap event signer.

If no settled BOLT12 payer proof is available, the event MUST NOT be counted as a BOLT12 zap.

## Counting Zaps

To compute the verified cumulative zap amount for an event, clients query for kind `9737` events with `#e` equal to the event id. For addressable events, clients query by `#a`. For profile zaps, clients query by `#p` and ignore events that also contain `e` or `a`.

Clients MUST validate each zap event before counting it.

Clients MUST deduplicate zap events for the same target by proof identifier. The proof identifier is the hash of the BOLT12 payer proof, or another stable proof identifier defined by the payer-proof specification.

If two valid zap events for the same target have the same proof identifier, clients MUST count only one. If amounts differ, clients MUST count the lower amount.

The cumulative zap amount is a client-computed view over valid public proof events. It is not consensus.

## Privacy

BOLT12 zaps are public proof events. They reveal at least amount, target, recipient pubkey, event time, and relays used. If `P` is present, they also reveal payer identity.

Payers who want private payments SHOULD pay the BOLT12 offer without publishing a zap event. Private payments are not counted as public zaps.

BOLT12 blinded paths can hide the recipient's node id, but Nostr zap metadata still reveals social payment relationships.

## Non-Goals

This NIP does not define:

- LNURL zap callbacks;
- BOLT11 zap invoices;
- custodial zap servers;
- private zaps;
- on-chain zaps;
- Cashu or Nutzap receipts;
- anti-sybil scoring policy;
- relay-level aggregation;
- pending or claimed-but-unsettled zaps.

## Rationale

A separate zap request event is unnecessary when BOLT12 payer proofs can prove payment directly.

The zap commitment exists to bind the Lightning payment proof to the Nostr target. Without this binding, a payer could reuse proof from one payment to claim zaps on unrelated events.

The commitment is not the zap event id because the zap event contains the proof, which is only available after payment. A random `zap_id` plus a deterministic commitment over the target fields avoids that circular dependency.

Only settled payer proofs count. Counting pending async payment artifacts would let payers fake arbitrary zap amounts.

## Compatibility

NIP-102 does not replace NIP-57 immediately. Clients MAY display NIP-57 and NIP-102 receipts together, but they MUST validate and label them separately.

NIP-57 `zap` tags are not reused. Events SHOULD use `zap12` tags for BOLT12 zap splits to avoid ambiguity with LNURL zaps.
