# NIP-100

## Public BOLT12 Receive Discovery

`draft` `optional`

This NIP defines how a Nostr user can publish a public BOLT12 offer associated with their public Nostr identity.

This NIP only defines discovery: given a Nostr pubkey, a client can find the user's current public BOLT12 receive offer.

## Dependencies

Implementations use:

- NIP-01 for kind `0` metadata events.
- BOLT12 offers.

## Profile Metadata

A user who wants to publish a public BOLT12 receive offer MUST include an `lno` field in the JSON content of their latest valid kind `0` metadata event:

```json
{
  "name": "alice",
  "lno": "lno1..."
}
```

The `lno` value MUST be a raw BOLT12 offer.

## Lookup

To discover a user's public BOLT12 receive offer, a client SHOULD resolve the user's latest valid kind `0` metadata event using the same relay discovery strategy it uses for profile metadata.

If the latest valid metadata event contains a valid top-level `lno` value, the client MAY use that value as the user's current public BOLT12 receive offer.

If the latest valid metadata event has no `lno` field, or if the field is not a valid canonical raw BOLT12 offer, the client MUST treat the user as having no public BOLT12 receive offer under this NIP.

Kind `0` metadata is mutable. Clients SHOULD treat the latest valid `lno` value as the current profile-level receive offer.

Protocols that need durable event-specific payment targets, split recipients, or validation after offer rotation MUST pin the offer in their own events or tags.

## Privacy

Publishing `lno` links a Nostr identity to a public Lightning receive target.

Clients and wallets MUST NOT publish or update a user's `lno` field without explicit user consent.
