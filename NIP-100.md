# NIP-100

## Public Receive Payment Instruction Discovery

`draft` `optional`

This NIP defines how a Nostr user can publish a public BIP-321 receive payment instruction associated with their public Nostr identity.

This NIP only defines discovery: given a Nostr pubkey, a client can find the user's current public receive payment instruction.

## Dependencies

Implementations use:

- NIP-01 for kind `0` metadata events.
- BIP-321 payment instructions.
- BOLT12 offers.

## Profile Metadata

A user who wants to publish a public receive payment instruction MUST include a `bip321` field in the JSON content of their latest valid kind `0` metadata event:

```json
{
  "name": "alice",
  "bip321": "bitcoin:?lno=lno1..."
}
```

The `bip321` value MUST be a BIP-321 URI. It MUST contain an `lno` instruction whose value is a valid BOLT12 offer.

The BIP-321 URI MAY contain additional payment instructions. Clients MUST use only payment instructions they support.

## Lookup

To discover a user's public receive payment instruction, a client SHOULD resolve the user's latest valid kind `0` metadata event using the same relay discovery strategy it uses for profile metadata.

If the latest valid metadata event contains a valid top-level `bip321` value, the client MAY use that value as the user's current public receive payment instruction.

If the latest valid metadata event has no `bip321` field, or if the field is not a valid BIP-321 URI containing a valid BOLT12 `lno` instruction, the client MUST treat the user as having no public receive payment instruction under this NIP.

Kind `0` metadata is mutable. Clients SHOULD treat the latest valid `bip321` value as the current profile-level receive payment instruction.

Protocols that need durable event-specific payment targets, split recipients, or validation after rotation MUST pin the payment instruction in their own events or tags.

## Privacy

Publishing `bip321` links a Nostr identity to a public payment instruction and the receive targets it contains.

Clients and wallets MUST NOT publish or update a user's `bip321` field without explicit user consent.
